# Проект: Auracrafts — Подарки ко Дню Матери (США)

## 1. Основная информация
*   **Цель проекта:** Создание современного, адаптивного веб-сайта-каталога (аффилиат-маркетинг) на тему подарков ко Дню Матери в США. Основной источник товаров — Creative Fabrica.
*   **Домен (план):** `auracrafts.app-cn.net` (уточнить у пользователя).
*   **Целевая аудитория:** Жители США, ищущие идеи и материалы для подарков мамам (DIY, графика, вышивка, лазерная резка).
*   **Статус на март 2026:** Сайт должен выглядеть максимально современно, удобно и привлекательно, соответствуя трендам веб-дизайна 2026 года.

## 2. Технологический стек (Рекомендованный для максимальной производительности)
*   **Фреймворк:** **Astro 5** (рекомендуется для контентных сайтов, отличная производительность, поддержка островной архитектуры). Альтернатива: Next.js 15 (если нужна более сложная интерактивность). Окончательный выбор обсудить с пользователем.
*   **Стилизация:** **Tailwind CSS v4** (утилитарный CSS, идеально для быстрой разработки и кастомизации) + **CSS Modules** для сложных компонентов.
*   **Интерактивность:** **Alpine.js** или **htmx** для лёгких взаимодействий. Если потребуется мощная динамика — использовать **Vue.js** или **React** с интеграцией через острова Astro.
*   **Типизация:** **TypeScript** (строгий режим) для всего кода, включая компоненты и утилиты.
*   **Данные:** Хранить в **JSON** или **YAML** в папке `src/data/`. Для статей использовать **Markdown** с коллекциями контента Astro.
*   **Оптимизация изображений:** Встроенные возможности Astro (`@astrojs/image`) или сторонние решения (Sharp). Все изображения должны быть в современных форматах (WebP/AVIF) с автоматической генерацией разных размеров.
*   **Тестирование:** **Vitest** для модульных тестов, **Playwright** для e2e-тестирования основных пользовательских сценариев.
*   **Линтер/Форматтер:** **ESLint** (строгая конфигурация) + **Prettier** (автоматическое форматирование).
*   **Хостинг/Деплой:** **Cloudflare Pages** или **Netlify** (с поддержкой Edge Functions, если понадобятся).

## 4. Правила и рекомендации для разработки (Уровень: Senior / Архитектор)

### 4.1. Архитектура и проектирование
1.  **Обоснование решений:** Перед реализацией любой значимой функциональности предлагать архитектурный подход, обсуждать его с пользователем и фиксировать ключевые решения (можно в `DECISIONS.md`).
2.  **Принципы SOLID, KISS, DRY, YAGNI:** Соблюдать неукоснительно. Компоненты должны быть слабо связаны, легко тестируемы и переиспользуемы.
3.  **Масштабируемость:** Проектировать так, чтобы добавление новых разделов или типов контента требовало минимум изменений в коде. Использовать коллекции контента Astro и динамические маршруты.
4.  **Управление состоянием:** Для глобального состояния (если потребуется) использовать легковесные решения (nano stores из Astro, Zustand). Избегать сложных стейт-менеджеров без необходимости.

### 4.2. Качество кода
1.  **Чистота и читаемость:** Код должен быть самодокументируемым. Комментарии допустимы только для сложных бизнес-правил или неочевидных решений.
2.  **Именование:** Использовать осмысленные имена переменных, функций и компонентов. Для CSS-классов следовать Tailwind, но при необходимости — BEM.
3.  **Типизация:** Строгий TypeScript (запрет `any`, использование `unknown` там, где тип не определён). Все данные, приходящие из JSON, должны иметь интерфейсы.
4.  **Линтинг и форматирование:** Настроить ESLint с правилами для TypeScript и Prettier для автоматического форматирования. Проверять код перед каждым коммитом (husky + lint-staged).
5.  **Декомпозиция:** Любой компонент, превышающий 200 строк, должен быть разбит на более мелкие. Повторяющаяся логика выносится в утилиты или хуки.

### 4.3. Производительность (Core Web Vitals)
1.  **Изображения:** Обязательно использовать оптимизацию (Astro `<Image />` или `<Picture />`). Форматы WebP/AVIF, разные размеры для разных экранов (srcset). Lazy-loading для изображений ниже сгиба.
2.  **JavaScript:** Минимизировать объём JS. Использовать островную архитектуру Astro (интерактивные компоненты загружаются только при необходимости). Критический CSS инлайнить.
3.  **Шрифты:** Использовать современные переменные шрифты (например, Inter, Onest) с оптимизацией (`font-display: swap`), подгружать только нужные веса.
4.  **Кэширование:** Настроить заголовки кэширования на уровне хостинга. Для статики использовать долгосрочное кэширование с хешами в именах файлов.
5.  **Мониторинг:** Внедрить измерение реальных Core Web Vitals (например, через `web-vitals` библиотеку) и анализировать результаты.

### 4.4. Доступность (a11y) и UX
1.  **Семантика:** Использовать правильные HTML5-элементы (`<header>`, `<main>`, `<article>`, `<section>`, `<nav>`, `<button>` и т.д.).
2.  **ARIA:** Где необходимо, добавлять ARIA-атрибуты (роли, метки, описания) для улучшения навигации с помощью скринридеров.
3.  **Клавиатурная навигация:** Все интерактивные элементы должны быть доступны с клавиатуры (фокус, tabindex, обработчики Enter/Space).
4.  **Контрастность:** Соблюдать минимальный контраст текста (4.5:1 для обычного текста, 3:1 для крупного). Проверять инструментами разработчика.
5.  **Mobile-first:** Разрабатывать сначала для мобильных устройств, затем адаптировать для планшетов и десктопов. Убедиться, что все кнопки имеют достаточный размер (минимум 44x44 пикселя).

### 4.5. SEO и метаданные
1.  **Уникальные мета-теги:** Каждая страница должна иметь уникальные `<title>`, `<meta name="description">`. Использовать коллекции контента для автоматической генерации.
2.  **Open Graph:** Генерировать og:title, og:description, og:image (изображение страницы или логотип) для шаринга в соцсетях.
3.  **JSON-LD (Schema.org):** Добавлять структурированные данные для товаров (Product), статей (Article), страниц (WebPage). Это улучшит отображение в поиске.
4.  **ЧПУ (Slug):** URL страниц должны быть читаемыми и содержать ключевые слова (например, `/gift-ideas/wooden-photo-frame`).
5.  **Sitemap:** Автоматически генерировать sitemap.xml при билде.

### 4.6. Работа с аффилиат-ссылками
1.  **Формирование ссылок:** Базовая ссылка: `https://www.creativefabrica.com/ref/20531415/`. Для конкретных товаров можно добавлять путь (например, `/product/123/`) и параметр ref. Уточнить у пользователя механику.
2.  **Атрибуты:** Все исходящие аффилиат-ссылки должны иметь `rel="sponsored nofollow"` и `target="_blank"` (чтобы не терять посетителя).
3.  **Текст кнопки:** "Click here and download" — чёткий призыв к действию. Кнопка должна быть хорошо заметна (контрастный цвет, достаточный padding).
4.  **Кликабельность картинок:** Сама картинка товара также должна вести на аффилиат-ссылку (улучшает UX).

### 4.7. Работа с данными (Creative Fabrica)
1.  **Источник данных:** Необходимо обсудить с пользователем, как именно будут собираться данные о товарах. Возможные варианты: ручной сбор в JSON, использование неофициального API, парсинг (требует отдельного обсуждения этичности и легальности).
2.  **Структура JSON:** Определить интерфейс Product:
    ```typescript
    interface Product {
      id: string;
      name: string;
      description: string;
      imageUrl: string; // локальный путь к изображению после загрузки
      affiliateUrl: string; // полная аффилиат-ссылка
      category: 'laser-cutting' | 'graphics' | 'embroidery';
    }
Загрузка изображений: Если данные собираются автоматически, предусмотреть скрипт, который скачивает изображения локально, оптимизирует их и сохраняет в public/images/, а в JSON записывает локальный путь. Это критически важно для производительности.

Валидация: Проверять входящие данные на корректность (наличие обязательных полей, правильность URL).

4.8. Тестирование и надёжность
Модульные тесты: Покрыть тестами ключевые утилиты (форматирование дат, фильтрация товаров, создание ссылок). Использовать Vitest.

Интеграционные тесты: Для страниц, которые динамически генерируются (например, список товаров), проверить, что они правильно отображаются с тестовыми данными.

E2E-тесты: Написать несколько критических сценариев (переход по меню, клик по карточке товара, проверка наличия аффилиат-ссылки) с помощью Playwright.

Кросс-браузерность: Проверить отображение в последних версиях Chrome, Firefox, Safari (включая мобильные). Использовать инструменты разработчика для эмуляции.

4.9. Процесс разработки и коммуникация
Итеративный подход: Разбивать проект на небольшие этапы (спринты). Например:

Этап 1: Настройка проекта, базовый макет, главная страница.

Этап 2: Компонент карточки товара, JSON-данные.

Этап 3: Страницы разделов, динамическая маршрутизация.

Этап 4: Блог и идеи для подарков.

Этап 5: Финальная полировка, SEO, тестирование.

Демонстрация результата: После каждого этапа показывать работающий прототип пользователю (например, через deploy preview на Netlify/Vercel) и собирать обратную связь.

Документирование: Все ключевые архитектурные решения фиксировать в DECISIONS.md (или в этом файле). Это поможет в будущем поддерживать проект.
